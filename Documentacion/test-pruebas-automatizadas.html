Documentación de Pruebas Automatizadas

Excel Manager - MVP de Visualización de Estado de Archivos

Introducción

Este documento describe las pruebas automatizadas implementadas para el MVP de visualización de estado de archivos Excel. Las pruebas están diseñadas para verificar la correcta funcionalidad e integración de los diferentes componentes del sistema, así como para validar el sistema de roles y permisos.

Las pruebas se han organizado en tres categorías principales:

  • Pruebas Unitarias: Verifican el funcionamiento correcto de componentes individuales
  • Pruebas de Integración: Comprueban la interacción entre diferentes componentes
  • Pruebas de Roles y Permisos: Validan el sistema de control de acceso

Estructura de Pruebas

Archivos y Ubicaciones

Las pruebas automatizadas se encuentran en los siguientes archivos:

  • src/scripts/run-unit-tests.js - Pruebas unitarias
  • src/scripts/run-integration-tests.js - Pruebas de integración
  • src/scripts/test-roles-permissions.js - Pruebas de roles y permisos
  • src/scripts/run-all-tests.js - Script unificado para ejecutar todas las pruebas

Comandos de Ejecución

Las pruebas pueden ejecutarse mediante los siguientes comandos npm:

# Ejecutar todas las pruebas
npm run test

# Ejecutar solo pruebas unitarias
npm run test:unit

# Ejecutar solo pruebas de integración
npm run test:integration

# Ejecutar solo pruebas de roles y permisos
npm run test:roles

Resultados de Pruebas

Los resultados de las pruebas se almacenan en la carpeta test-results/ en los siguientes formatos:

  • Archivos JSON con resultados detallados de cada tipo de prueba
  • Un archivo HTML con un informe visual unificado de todas las pruebas

Pruebas Unitarias

Las pruebas unitarias se centran en verificar el funcionamiento correcto de los controladores de forma aislada, simulando las solicitudes HTTP y validando las respuestas.

Objetivos

  • Verificar que los controladores funcionan correctamente con diferentes parámetros
  • Comprobar que las respuestas tienen la estructura esperada
  • Validar el manejo adecuado de errores

Casos de Prueba

getFileStatusData - agrupación por carpeta Exitoso

Verifica que el controlador devuelve correctamente los datos agrupados por carpeta.

  • Parámetros: { groupBy: 'folder' }
  • Verificación: La respuesta tiene estructura correcta con datos agrupados
getFileMetrics - timeFrame week Exitoso

Verifica que el controlador devuelve métricas filtradas por la última semana.

  • Parámetros: { timeFrame: 'week' }
  • Verificación: Datos filtrados por fecha y estructura correcta de stats/puzzleItems
updateFileStatus - estado inválido Exitoso

Verifica que el controlador rechaza correctamente un estado no válido.

  • Parámetros: { status: 'estadoInvalido' }
  • Verificación: Error 400 con código 'INVALID_STATUS'

Pruebas de Integración

Las pruebas de integración verifican la interacción entre diferentes componentes del sistema, asegurando que todos funcionen correctamente en conjunto.

Objetivos

  • Verificar la comunicación entre la API y la base de datos
  • Comprobar que los datos se procesan correctamente a través de múltiples capas
  • Validar la consistencia de datos entre MongoDB y la API

Casos de Prueba

Endpoint de estado de archivos Exitoso

Verifica que el endpoint de estado de archivos responda correctamente.

  • Endpoint: /api/files/status
  • Verificación: Respuesta 200 con estructura adecuada
Cambio de estado de archivo Exitoso

Verifica que se pueda cambiar el estado de un archivo a través de la API y que la base de datos se actualice correctamente.

  • Endpoint: /api/files/:id/status
  • Verificación: Cambio persistido en MongoDB
Consistencia entre MongoDB y API Exitoso

Verifica que el número de archivos en MongoDB coincida con el reportado por la API.

  • Endpoint: /api/files/metrics
  • Verificación: Conteo de documentos en MongoDB = totalFiles en API

Pruebas de Roles y Permisos

Las pruebas de roles y permisos validan el sistema de control de acceso, verificando que solo los usuarios autorizados puedan acceder a ciertas funcionalidades.

Objetivos

  • Verificar que los roles tengan los permisos adecuados
  • Comprobar que los usuarios solo pueden acceder a los recursos permitidos
  • Validar que la estructura de datos soporte permisos por área y subárea

Casos de Prueba

Acceso como administrador Exitoso

Verifica que un usuario con rol de administrador pueda acceder a los endpoints de estado de archivos.

  • Endpoint: /api/files/status
  • Verificación: Acceso permitido (código 200)
Modelos compatibles con permisos Exitoso

Verifica que los modelos tengan los campos necesarios para soportar el sistema de permisos.

  • Verificación: Campos areaId, subareaId, companyId presentes en modelos File y User
Permisos de archivos Parcial

Verifica la existencia de permisos específicos para la gestión de archivos.

  • Verificación: Existencia de permisos file.read, file.write, file.delete

Interpretación de Resultados

Al ejecutar las pruebas, se generará un informe detallado que muestra:

  • Resumen general: Número total de pruebas, pruebas exitosas, pruebas fallidas y tasa de éxito
  • Resultados detallados: Para cada tipo de prueba, detalles sobre cada caso de prueba
  • Tiempos de ejecución: Cuánto tardó en ejecutarse cada conjunto de pruebas

Interpretando Fallos

Cuando una prueba falla, el informe mostrará:

  • Nombre de la prueba fallida
  • Mensaje de error detallado
  • Datos adicionales relevantes para diagnosticar el problema

Los fallos más comunes suelen ser:

  • Inconsistencia de estructura: La estructura de datos no coincide con lo esperado
  • Fallos de conexión: Problemas al conectar con la base de datos o la API
  • Problemas de permisos: Problemas con el sistema de roles y permisos

Ejecución Manual vs Automatizada

Aspecto Pruebas Manuales Pruebas Automatizadas
Tiempo de ejecución Minutos a horas Segundos a minutos
Consistencia Variable (depende del tester) Alta (siempre igual)
Cobertura Limitada por tiempo/recursos Amplia y configurable
Detección de regresiones Difícil de mantener Inmediata con cada ejecución
Documentación Requiere esfuerzo adicional Auto-documentado en código

Las pruebas automatizadas complementan las pruebas manuales, pero no las reemplazan completamente. Ciertas áreas como la experiencia de usuario y la apariencia visual todavía requieren verificación manual.

Próximos Pasos

Para mejorar aún más el sistema de pruebas, se recomiendan los siguientes pasos:

  1. Aumentar cobertura: Añadir más casos de prueba para cubrir más funcionalidades
  2. Pruebas End-to-End: Implementar pruebas que verifiquen flujos completos de usuario con Cypress o Playwright
  3. Integración con CI/CD: Ejecutar pruebas automáticamente en cada integración de código
  4. Pruebas de carga: Verificar el rendimiento con grandes volúmenes de datos
  5. Informes visuales mejorados: Mejorar los informes de pruebas con gráficos y tendencias
